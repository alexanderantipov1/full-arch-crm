# Q1 — ENG-182 Identity Match Candidate Foundation

## Status

Complete.

## Summary

Added the explicit `identity.match_candidate` decision ledger from the P2
data-foundation plan as the first additive slice of ENG-182. The new table
records every cross-provider identity matching decision (open ambiguity,
auto-accept, accept, reject, supersede) with a tenant-scoped uniqueness shape
that mirrors what the future pipeline (ENG-183 / ENG-185) will require to
replace the hidden email/phone reactivation logic currently inside
`SfLeadIngestService`.

No existing migration was edited. No ingest behavior, route, or worker
changed. The implementation respects every cross-package import rule in
`packages/CLAUDE.md` (identity does not import ingest, so `hint_id` stays a
bare UUID column with no Python-level FK to the not-yet-existing
`ingest.normalized_person_hint`).

## Deviations From The P2 Proposal

1. **`hint_id` stays bare UUID (no FK / no relationship).** P2 §
   "identity.match_candidate" notes "plain UUID pointer to
   `ingest.normalized_person_hint.id`; avoid identity importing ingest".
   This implementation honors that: the column is plain, and the migration
   omits the FK because the hints table does not exist yet. The follow-up
   migration that lands `ingest.normalized_person_hint` (ENG-183) can add
   the FK then if desired.

2. **CHECK constraint names use the project naming convention.** The base
   `NAMING_CONVENTION` (`packages/db/base.py`) wraps `ck_` constraints as
   `ck_<table>_<name>` — so the P2 `CHECK status IN (...)` becomes
   `ck_match_candidate_status` after the `op.f()` pass. Same for the other
   three CHECKs.

3. **Indexes are tenant-scoped composites.** Where P2 listed e.g.
   `ix_match_candidate_candidate` on `(tenant_id, candidate_person_uid)`,
   I kept the same shape, which matches how every other ENG-128 per-tenant
   table indexes lookup paths. `ix_match_candidate_tenant_id` covers
   tenant-wide scans separately.

4. **Service-layer PHI guard is a deny-list, not allow-list.** The P2 plan
   says evidence/conflicts must be non-clinical but does not specify the
   exact schema yet. Until product settles canonical evidence keys
   (ENG-183), the service rejects a conservative deny-list of likely PHI
   field names (`dob`, `allergies`, `medication`, `prescription`,
   `clinical_notes`, `raw_payload`, etc.). The list lives in
   `packages/identity/service.py::_FORBIDDEN_EVIDENCE_KEYS` and is easy to
   widen as feedback comes in.

5. **`decided_at` set in the service, not by a CHECK constraint.** P2
   listed `decided_at` as "set for all non-`open` statuses". I made the
   service stamp `decided_at` automatically when status is not `open`,
   rather than enforcing it via a DB CHECK — this keeps the invariant
   live without requiring callers to remember to set the column, and is
   easy to add a DB-level CHECK later if drift is observed.

6. **No state-transition service methods this slice.** P2 listed
   `find_open_match_for_pair(...)`, `mark_match_decided(...)`,
   `list_candidate_persons_by_identifiers(...)`. Q1 ships
   `add_match_candidate`, `find_open_match_for_pair`, and a few
   repository helpers; the matchmaking helpers and state-transition
   methods (`auto_accept_match`, `evaluate_match_policy`,
   `resolve_or_create_from_hint`) belong in the next wave that wires the
   pipeline (ENG-185). Reason: this slice is deliberately additive +
   data-foundation only.

## Follow-up Tasks

- **ENG-183 — ingest.normalized_person_hint table.** Once that lands,
  add the FK from `identity.match_candidate.hint_id` to
  `ingest.normalized_person_hint.id` in a small additive migration.
  Keep the column plain UUID in the model layer to preserve the
  identity-does-not-import-ingest rule.
- **ENG-185 — pipeline integration.** Move the email/phone reactivation
  matching out of `SfLeadIngestService._resolve_person` and into a new
  `IdentityService.resolve_or_create_from_hint(...)`. That method should
  call `add_match_candidate(...)` for every decision, including no-op
  auto-accept rows (so the ledger is complete).
- **Canonical evidence keys.** Replace the deny-list
  `_FORBIDDEN_EVIDENCE_KEYS` with an allow-list of normalized signal
  names (e.g. `email_match`, `phone_match`, `name_compatibility`) when
  product agrees on the shape. Track under ENG-183.
- **DB-level integration tests.** The model-level tests verify metadata.
  Once a Postgres test container fixture is wired in
  (`tests/integration/`), add tests that exercise the partial-unique
  guards (two open candidates for the same pair must fail, etc.) and
  the CHECK constraints.
