## Summary

Plan the smallest safe ENG-185 patch that replaces the hidden email/phone
reactivation logic currently embedded in
`packages/ingest/sf_lead_service.py::SfLeadIngestService._resolve_person`
with an explicit `IdentityService.resolve_or_create_from_hint(...)` entry
point. The new entry point consumes the
`ingest.normalized_person_hint` row produced by R1 / ENG-185, applies the
P2 §"Match Policy" tier ladder, writes an `identity.match_candidate`
ledger row for every non-trivial decision, and returns a typed result so
the Salesforce pipeline (and the future CareStack pipeline) can populate
`ops.lead` / `ops.inquiry` without owning matching policy.

This plan assumes R1 has not yet landed (R1 report is absent at write
time). Where R1 shape matters, the dependency is called out below.

## Current Flow Summary

### Where the hidden match logic lives today

`packages/ingest/sf_lead_service.py` lines 105–229:

1. `SfLeadIngestService.pull_recent(...)` iterates SOQL records, calls
   `IngestService.capture(...)` (verbatim `raw_event` row) before any
   mapping, then calls `_resolve_person(...)` and `OpsService.upsert_lead(...)`.
2. `_resolve_person(tenant_id, sf_id, record)` runs the dedupe ladder:
   - **(a)** `IdentityService._repo.find_source_link(tenant_id, "salesforce", "lead", sf_id)`
     — re-pull → return existing person, `is_reactivation=False`.
   - **(b)** If `record["Email"]` present, `IdentityService.resolve_by_email(...)` →
     on hit call `IdentityService.add_source_link(...)` and return
     `is_reactivation=True`.
   - **(c)** If `record["Phone"]` present, `IdentityService.resolve_by_phone(...)` →
     on hit call `IdentityService.add_source_link(...)` and return
     `is_reactivation=True`.
   - **(d)** Fallback: `IdentityService.resolve_or_create_person(tenant_id,
     "salesforce", "lead", sf_id, hints=PersonIn(...))` creates a new
     person + source_link with identifiers from the SF record, returns
     `is_reactivation=False`.

Tests for this ladder are in
`tests/ingest/test_sf_lead_service.py` (5 mock-based branch tests + 2
limit/SOQL tests + 1 `list_recent` mapping test) — every one of them
mocks the four `IdentityService` calls above.

### Boundary observations

- Line 180 reaches across the service boundary into
  `self._identity._repo.find_source_link(...)`. The new entry point lets
  ingest stay above the repository.
- `ingest` is allowed to import `identity` (matrix in
  `packages/CLAUDE.md`). `identity` is NOT allowed to import `ingest`.
  The `MatchCandidate.hint_id` column already lives as a plain UUID
  pointer for this reason (Q1 report §"Deviations" item 1).
- `SfLeadIngestService` only consumes
  `IdentityService.{resolve_by_email, resolve_by_phone, add_source_link,
  resolve_or_create_person, get_person}` plus the protected `_repo`
  call. After ENG-185 it should consume one method:
  `IdentityService.resolve_or_create_from_hint(...)`.
- No worker job exists for Salesforce yet (`apps/worker/jobs/` has no
  SF entry). The only invocation path today is the API route
  `apps/api/routers/integrations.py::pull_recent`.

## Proposed ENG-185 Implementation Steps

### Step 1 — Identity-only: new entry point skeleton + DTO

Owned write scope:

- `packages/identity/service.py` — add `resolve_or_create_from_hint(...)`
  with the source-link-only tier wired and the high-confidence /
  open-candidate tiers stubbed.
- `packages/identity/schemas.py` — add `ResolveFromHintResult` DTO (see
  "Contract changes" below).
- `packages/identity/repository.py` — add
  `list_candidate_persons_by_identifiers(tenant_id, email, phone)`.
- `tests/identity/test_resolve_or_create_from_hint.py` — new test
  module.
- `packages/identity/CLAUDE.md` — document the new entry point as the
  provider entry point for ingest workers.

The skeleton should call the existing `resolve_or_create_person(...)`
for the source-link-exact case so behavior is unchanged when the hint
matches an existing `(source_system, source_kind, source_id)` triple.

No ingest, no SF service, no Alembic revision in this step.

### Step 2 — Identity-only: tier-1 auto-accept + tier-2 open candidate

Owned write scope (same as step 1, no migration):

- `packages/identity/service.py` — add private helpers
  `_evaluate_match_policy(hint, candidates)` (pure function) and
  `_apply_auto_accept(hint, person, match_rule, confidence, evidence)`
  that writes the source_link and the `MatchCandidate(status='auto_accepted')`
  row in one unit-of-work.
- A `MatchPolicyConfig` constants block (thresholds 0.99 / 0.95 / 0.92
  from P2 §"Match Policy") that can be tuned without touching call
  sites.
- Tests cover the full tier ladder:
  - exact source link → no match candidate, person reused
    (tier 0);
  - email+phone+compatible name → `auto_accepted` + new source_link
    on existing person (tier 1);
  - exact phone, name compatible → `auto_accepted` (tier 1);
  - exact email + name compatible, no competing phone → `auto_accepted`
    (tier 1);
  - email only with multiple candidates → `open` + a new
    source-linked person so the pull keeps moving (tier 2);
  - cross-tenant candidate must NOT leak into matches;
  - re-call with same `hint_id` is idempotent (uses the
    `uq_match_candidate_hint_candidate_active` partial unique guard);
  - bad evidence keys still blocked by existing PHI deny-list.

### Step 3 — Ingest pipeline cutover

Owned write scope:

- `packages/ingest/sf_lead_service.py` — rewrite `_resolve_person(...)`
  to:
  1. call `IngestService.capture_normalized_person_hint(...)` (from R1)
     with the SF record fields already exposed to identifier
     normalization (`Email`, `Phone`, names, `Id`);
  2. call `IdentityService.resolve_or_create_from_hint(...)` with the
     resulting hint DTO;
  3. map the typed result into `(person, is_reactivation)` for the
     existing `OpsService.upsert_lead` call.
- `tests/ingest/test_sf_lead_service.py` — rewrite to mock the new
  identity entry point (one method) instead of the four prior calls.
  Keep the existing branch coverage: re-pull / exact source link /
  high-confidence email match / high-confidence phone match / ambiguous
  open match / brand-new person.
- No changes to the API route, the SOQL string, the DTO shape, or
  `ops.lead` storage. `is_reactivation` semantics preserved: any hint
  that resolves to an existing person via match policy (not via
  source-link exact) is a "reactivation".

### Step 4 (optional, additive migration) — FK from match_candidate.hint_id

Owned write scope:

- `packages/db/alembic/versions/<new>_add_match_candidate_hint_fk.py`

## Risks

### PHI / raw payload

- `IdentityService` must NOT accept a raw provider record. The new
  entry point receives a `NormalizedPersonHint` DTO (or, in step 1's
  skeleton, a typed `MatchHintIn` with just email/phone/name/source
  triple). The SF service must NOT pass the SOQL record into identity.
- The existing `_FORBIDDEN_EVIDENCE_KEYS` deny-list (recursive) in
  `packages/identity/service.py::_reject_phi_keys` already guards
  `MatchCandidate.evidence` and `conflicts`. The new auto-accept path
  must still funnel evidence dicts through this guard.
- `ingest.normalized_person_hint.meta` / `quality_flags` are
  parser/matching metadata only; R1 brief explicitly enforces that. The
  identity service should defensively reject hints whose `meta` carries
  PHI-shaped keys, even though R1 catches them upstream — defense in
  depth, same deny-list.

### Tenant isolation

- All `IdentityRepository` reads run through `for_tenant(...)`. The new
  `list_candidate_persons_by_identifiers(...)` must also filter by
  `tenant_id` on the `PersonIdentifier` join, with no cross-tenant
  fallback.
- The Q1 review fix already requires
  `IdentityRepository.get_person(tenant_id, person_uid)` on every
  person reference inside `add_match_candidate`. The auto-accept path
  must reuse the same checks before writing the source_link to a
  candidate person.

### Idempotency

- Re-pull of the same SF Lead Id must produce:
  - no new `ingest.normalized_person_hint` row (R1 enforces
    `UNIQUE (tenant_id, raw_event_id)`); on duplicate the service
    returns the existing hint;
  - no new `identity.source_link` row (existing
    `uq_source_link_external` covers this);
  - no new `identity.match_candidate` row in the auto-accepted /
    accepted / open buckets for the same `(tenant_id, hint_id,
    candidate_person_uid)` triple (existing
    `uq_match_candidate_hint_candidate_active` partial unique covers
    this).
- The new entry point must NOT write a match_candidate for the
  source-link-exact path (P2 §"Tier 0"). A trivially-resolved re-pull
  should leave the ledger untouched so the table doesn't bloat.

### Source-link behavior

- Today's tier-(b)/(c) email/phone reactivation calls
  `IdentityService.add_source_link(tenant_id, matched.id, "salesforce",
  "lead", sf_id)`. The new auto-accept must do the same in the same
  unit of work and BEFORE writing the match_candidate row, so that if
  the auto-accept commit succeeds the source_link is definitely in
  place. Order matters: the partial unique guard on the candidate row
  treats accepted-status rows the same as auto-accepted, so a partial
  failure could leave a candidate without a source_link if the order
  flipped.
- The auto-accept case has `source_person_uid=None` (no new person was
  ever created), `candidate_person_uid=<existing>`, and
  `accepted_person_uid=<same existing>`. This is supported by the
  current model + service — confirmed in
  `test_add_match_candidate_checks_all_person_refs_in_tenant` in
  `tests/identity/test_match_candidate_service.py`.

### `is_reactivation` flag continuity

- Today `ops.lead.extra.is_reactivation` is `True` whenever the dedupe
  ladder hit tier (b) or (c). After ENG-185 the semantic should be:
  "the hint resolved to a pre-existing person via match policy, not
  via exact source-link recapture". The new
  `ResolveFromHintResult.was_existing_person_match` (or equivalent
  field) is the post-ENG-185 source of truth.
- Tier 0 (exact source link) keeps `is_reactivation=False` (re-pull).
- Tier 1 auto-accept keeps `is_reactivation=True`.
- Tier 2 open with new source-linked person keeps
  `is_reactivation=False` — same as today's tier (d). Open
  candidates are not "reactivations" by themselves; an operator
  decision later may change the meaning, but the lead row should not
  retroactively flip on its own.

### Open candidates blocking pull

- P2 §"Tier 2: Open Candidate": ambiguous matches MUST NOT block the
  pull. The new entry point must still return a usable
  `(person, was_reactivation)` pair so the SF service writes the
  `ops.lead` row. Implementation: create a new source-linked person
  (current tier (d) behavior) + write
  `MatchCandidate(status='open')` with the conflict evidence. Operator
  reconciliation later supersedes or accepts.

### Backward compatibility with existing tests

- `tests/ingest/test_sf_lead_service.py` currently makes assertions
  against `service._identity.resolve_by_email`,
  `service._identity.resolve_by_phone`, and
  `service._identity._repo.find_source_link`. Step 3 rewrites these to
  assert against `service._identity.resolve_or_create_from_hint`
  (single mock) plus a hint-capture assertion against
  `service._ingest.capture_normalized_person_hint`. Branch coverage
  must remain at parity (re-pull, email-match auto-accept,
  phone-match auto-accept, ambiguous-open, brand-new).
- `tests/api/test_integrations_salesforce.py` does NOT inspect the
  matching layer — only the HTTP-level service dependency. No
  changes required at the API layer.

## Acceptance Criteria For ENG-185

1. `SfLeadIngestService._resolve_person` calls one identity method:
   `resolve_or_create_from_hint(...)`. It no longer touches
   `resolve_by_email`, `resolve_by_phone`, or `_repo.find_source_link`.
2. Every SF Lead pull writes exactly one
   `ingest.normalized_person_hint` row per `raw_event`.
3. A high-confidence Salesforce hint that matches an existing person
   writes one `identity.match_candidate(status='auto_accepted')` row,
   adds the new `salesforce/lead/<sf_id>` source_link to that person,
   and leaves `ops.lead.extra.is_reactivation=True`.
4. An ambiguous email-only match writes
   `identity.match_candidate(status='open')` with conflict evidence,
   creates a new person + source_link, leaves
   `ops.lead.extra.is_reactivation=False`, and returns successfully
   (does not raise, does not block the pull).
5. Re-pull of the same `(salesforce, lead, sf_id)`:
   - touches `last_seen_at` on the existing source_link;
   - does NOT write a new hint;
   - does NOT write a new match_candidate;
   - does NOT flip `is_reactivation` on the existing `ops.lead`.
6. CareStack pipeline (future ENG-184 / ENG-186) can call the same
   `resolve_or_create_from_hint(...)` without any change in the
   identity layer.
7. `make verify` clean; `alembic check` exits 0; `git diff --check`
   exits 0.
8. New `tests/identity/test_resolve_or_create_from_hint.py` covers
   every tier branch and the cross-tenant guard. Rewritten
   `tests/ingest/test_sf_lead_service.py` retains the previous branch
   coverage.

## Blockers / Dependencies On R1

R1 is not yet present in `reports/`. The dependency surface is:

1. **`ingest.normalized_person_hint` table + ORM model + DTOs** — R1
   ships it. ENG-185 reads `hint.id`, `hint.email_normalized`,
   `hint.phone_normalized`, `hint.given_name`, `hint.family_name`,
   `hint.display_name`, `hint.source_system`, `hint.source_kind`,
   `hint.source_id`, `hint.raw_event_id`, `hint.tenant_id`. If R1
   renames or drops any of these, step 1's `MatchHintIn` DTO needs to
   adapt.
2. **`IngestService.capture_normalized_person_hint(...)` method** —
   R1 ships it. SF service calls it; the SF service does NOT build
   the hint itself (avoids duplicate normalization logic).
3. **`IngestRepository.find_hint_by_raw_event(...)`** — needed only
   for the re-pull idempotency assertion. If R1 omits it, ENG-185
   either adds it (and PR B widens its write scope into ingest
   repository) or relies solely on the `UNIQUE (tenant_id,
   raw_event_id)` constraint catching the duplicate insert. The
   former is cleaner.
4. **Optional `raw_event` additive columns** (`payload_sha256`,
   `processing_key`, `source_observed_at`, `sync_run_id`) — P2 §
   "Optional same-PR `ingest.raw_event` additive columns" lists
   them. Whether R1 ships them is up to R1's worker. ENG-185 does
   not require them but the SF pipeline can populate
   `payload_sha256` once present to harden replay protection.
5. **Hint enum validation** — R1 brief allows local source-list
   validation. If R1 ships a check constraint with a list that
   differs from `identity.SOURCE_SYSTEMS` / `SOURCE_KINDS`, ENG-185
   adds a documented mapping in the SF pipeline. Simpler: keep
   both lists in lockstep; document the convention in R1's CLAUDE.md.

If any of (1)–(3) deviates from the P2 contract, PR A still ships
unchanged (it does not import ingest), but PR B (and its tests)
needs a small adapter and the report must note the drift.

## Contract Changes Needed Before Implementation

These should be agreed by Codex before PR A starts.

### 1. `ResolveFromHintResult` DTO

Lives in `packages/identity/schemas.py`. Minimum shape:

```python
class ResolveFromHintResult(BaseModel):
    model_config = ConfigDict(from_attributes=False)

    person_uid: UUID
    was_new_person: bool           # True only for tier-2-with-new-person + brand-new
    was_existing_person_match: bool  # True for tier-1 auto-accept
    was_source_link_recapture: bool  # True for tier-0 exact source-link
    match_candidate_id: UUID | None  # None for tier 0; set for tiers 1 and 2
    open_candidate: bool             # True only when status='open' was written
```

`was_existing_person_match` is the canonical replacement for today's
ad-hoc `is_reactivation`. The SF service maps it directly to
`ops.lead.extra.is_reactivation`.

### 2. `MatchHintIn` DTO (only if step 3 needs an adapter)

If R1's `NormalizedPersonHintOut` already has every field listed in
"Blockers/Dependencies On R1" item 1, identity service can accept the
R1 DTO directly. Otherwise, define a thin adapter in
`packages/identity/schemas.py` so identity does not depend on R1's
DTO module (preserves the "identity does not import ingest" rule
even at the DTO layer).

Recommendation: identity should accept a minimal `MatchHintIn` that
the SF service constructs from the R1 DTO. Keeps the cross-package
dependency one-way and easy to reason about.

### 3. Match policy thresholds

Initial values from P2 §"Match Policy":

- `email + phone + name compatible`: 0.99 → auto_accept.
- `phone + name compatible`: 0.95 → auto_accept.
- `email + name compatible, no phone conflict`: 0.92 → auto_accept.
- `phone-only` or `email-only` without name: below auto threshold
  (e.g. 0.70 floor) → open.

Encode as a `MatchPolicyConfig` dataclass or module-level constants
in `packages/identity/service.py`. Codex review should confirm the
floor value (0.70 is a placeholder until product weighs in).

### 4. Source-link write inside auto-accept

Recommend: the auto-accept path inside `IdentityService` writes the
`source_link` itself before writing the `MatchCandidate`. Alternative
(returning a "caller please add the source_link" hint) is rejected
because it would let two ingest pipelines drift on the exact
sequencing.

### 5. Match rule string taxonomy

The `match_rule` column on `match_candidate` is `String(64)`. Initial
taxonomy:

- `source_link` — tier 0 (never written, but reserved if we ever do
  log re-pulls).
- `email_phone_name` — tier 1 best case.
- `phone_name` — tier 1.
- `email_name` — tier 1.
- `email_only_ambiguous` — tier 2.
- `phone_only_ambiguous` — tier 2.

Codify in a `MATCH_RULES` tuple in `packages/identity/models.py`
alongside `MATCH_CANDIDATE_STATUSES`. Add a service-level validator
that the chosen rule is in the tuple.
