## Summary

Added the identity-only follow-up to ENG-185:
`IdentityService.resolve_or_create_from_hint(...)`, a provider-neutral
entry point that consumes a minimal identity-owned `MatchHintIn` DTO
(built by the ingest-side adapter from a captured
`ingest.normalized_person_hint` row), applies the P2 / R2 match policy
tier ladder, writes `identity.match_candidate` rows for non-trivial
decisions, and returns a typed `ResolveFromHintResult` for future
Salesforce / CareStack cutover.

This wave does NOT change Salesforce or CareStack ingest behavior.
`SfLeadIngestService` is untouched. No worker job changed. No API route
changed. No Alembic revision was added or edited. `packages.identity`
does not import `packages.ingest`; the new DTO is the contract.

## Implemented DTO / Result Contract

### `MatchHintIn`

Identity-owned, the contract the ingest adapter builds from
`ingest.normalized_person_hint` rows:

```python
class MatchHintIn(BaseModel):
    model_config = ConfigDict(from_attributes=True)

    hint_id: UUID | None = None
    source_system: str    # min 1, max 32; validated against SOURCE_SYSTEMS
    source_kind: str      # min 1, max 32; validated against SOURCE_KINDS
    source_id: str        # min 1, max 240 — required for source-link ops
    given_name: str | None
    family_name: str | None
    display_name: str | None
    email_normalized: str | None
    phone_normalized: str | None
    quality_flags: dict[str, object]
    meta: dict[str, object]
```

`source_id` is required (not nullable). All four tiers of the policy
either look up or write a `source_link`, and the existing
`resolve_or_create_person` already requires `source_id`. Making the
field required at the schema layer pushes the failure to the adapter
(where it has full context) instead of into the policy mid-flight.

### `ResolveFromHintResult`

```python
class ResolveFromHintResult(BaseModel):
    model_config = ConfigDict(from_attributes=False)

    person_uid: UUID
    was_new_person: bool = False
    was_existing_person_match: bool = False
    was_source_link_recapture: bool = False
    match_candidate_id: UUID | None = None
    open_candidate: bool = False
```

Service-enforced invariants:

- Exactly one of `was_source_link_recapture` /
  `was_existing_person_match` / `was_new_person` is `True` for Tier 0
  / Tier 1 / (Tier 2 ∪ Fallback) respectively.
- `open_candidate` is `True` only when `status='open'` was written,
  which by construction implies `was_new_person=True` (Tier 2 always
  creates a new source-linked person).
- `match_candidate_id` is `None` for Tier 0 and Fallback, and set for
  Tier 1 / Tier 2.

`was_existing_person_match` is the canonical post-ENG-185 replacement
for today's ad-hoc `ops.lead.extra.is_reactivation`.

## Match Policy Tier Behavior

| Tier | Trigger | Source link | Match candidate row | Result flags |
|---|---|---|---|---|
| 0 | `find_source_link((source_system, source_kind, source_id))` returns a row | `touch_source_link` only | none | `was_source_link_recapture=True` |
| 1 | exactly one candidate clears `email_phone_name@0.99` / `phone_name@0.95` / `email_name@0.92` | `add_source_link` (idempotent) BEFORE candidate row | `auto_accepted`; `candidate_person_uid` and `accepted_person_uid` both = matched person; `source_person_uid=None` | `was_existing_person_match=True`; `match_candidate_id` set |
| 2 | two or more candidates clear a tier-1 rule, OR one candidate matches but fails name / phone-conflict checks | `resolve_or_create_person(...)` creates a new source-linked person | `open`; `candidate_person_uid` = primary existing candidate; `source_person_uid` = new person; rule = `email_only_ambiguous` (default) or `phone_only_ambiguous` (when only phones matched) | `was_new_person=True`, `open_candidate=True`; `match_candidate_id` set |
| Fallback | no candidates returned by `list_candidate_persons_by_identifiers` | `resolve_or_create_person(...)` creates a brand-new source-linked person | none | `was_new_person=True` |

Name compatibility (`_names_compatible`):

- absence is not a conflict (if either side has no name tokens, the
  pair is treated as compatible);
- otherwise any case-insensitive overlap among
  `{given_name, family_name, display_name}` counts as compatible.

Phone conflict (blocks `email_name`):

- hint has a phone, candidate has at least one phone, the hint's
  phone is not among them.

Evidence / conflicts payloads:

- only normalized signal keys (`email_match`, `phone_match`,
  `name_compatible`, `candidate_count`, `email_match_any`,
  `phone_match_any`, `auto_accept_eligible`, `phone_conflict`); no
  raw provider fields;
- the existing recursive `_FORBIDDEN_EVIDENCE_KEYS` guard in
  `IdentityService.add_match_candidate` still applies (defense in
  depth);
- `resolve_or_create_from_hint` ALSO runs the recursive PHI guard on
  the incoming hint's `meta` and `quality_flags` before doing any
  DB work — so even if the ingest-side guard were to drift, the
  identity boundary catches it.

Idempotency:

- Tier 0 is naturally idempotent (it never writes a candidate row).
- Tier 1 / Tier 2 check `find_active_hint_candidate(tenant_id,
  hint_id, candidate_person_uid)` when `hint_id` is set; if an
  active (`open` / `auto_accepted` / `accepted`) row exists, return
  with `match_candidate_id=existing.id` and skip the second
  `add_source_link` / `add_match_candidate`.
- Without `hint_id`, the active-candidate lookup is skipped (the
  partial unique guard does not protect rows with `hint_id IS NULL`).

Cross-tenant isolation:

- `list_candidate_persons_by_identifiers` filters BOTH
  `Person.tenant_id` AND `PersonIdentifier.tenant_id` through
  `for_tenant`, so a foreign tenant's `PersonIdentifier` row with
  the same email cannot leak into the candidate pool.
- `add_match_candidate` continues to check every person reference
  via `IdentityRepository.get_person(tenant_id, person_uid)` (the Q1
  Codex review correction). The Tier 1 / Tier 2 paths construct
  their `MatchCandidateIn` payloads with persons resolved inside the
  same `tenant_id`, so the guard is the second line of defense.

## Deviations From R2 / P2

1. **No state-transition methods this wave.** R2 listed
   `auto_accept_match(...)` / `mark_match_decided(...)` as part of
   the broader ENG-185 surface. S1 ships ONLY
   `resolve_or_create_from_hint(...)`. Reason: the wave brief and
   Wave S contract scope to the entry point; state transitions land
   when the operator-review surface lands (no Linear ticket yet).

2. **`MatchHintIn.source_id` is required.** R2 noted that R1's
   `NormalizedPersonHintOut.source_id` is `str | None`. S1 makes the
   identity DTO's `source_id` required because every tier writes or
   touches a `source_link`, and `IdentityService.resolve_or_create_person`
   already requires `source_id`. The ingest adapter is the right
   place to drop hints with absent source IDs (they cannot be
   reactivated against a stable provider id anyway).

3. **`_OPEN_AMBIGUOUS_FLOOR = 0.70`.** R2 listed `0.70` as a
   placeholder. S1 ships it as the constant pending product input.
   The threshold lives in `packages.identity.service` as a
   module-level constant so it can be tuned in one place when
   product weighs in.

4. **Tier 2 rule selection.** R2's match-rule taxonomy lists
   `email_only_ambiguous` / `phone_only_ambiguous`. S1 picks
   `phone_only_ambiguous` when ONLY phones matched across the
   candidate set and `email_only_ambiguous` otherwise (including the
   both-matched case). Reason: the both-matched-but-ambiguous case
   is conceptually closer to an email-side ambiguity in our current
   data (Salesforce Leads carry email more reliably than phone).
   Easy to split into a third `email_phone_ambiguous` rule later.

5. **`_names_compatible` permissive on missing names.** R2 listed
   "compatible name" without a definition. S1 treats absence of name
   tokens as non-conflicting. Reason: provider rows frequently have
   no name on the new side (Salesforce Lead without a `LastName`,
   CareStack patient minimal record). Penalising absence would block
   most reactivations.

6. **No DB-level FK promotion of `match_candidate.hint_id`.** Q1
   left this as a follow-up after R1 lands. R1 has landed (in the
   working tree), but Wave S contract is migration-free, so the FK
   promotion is deferred to a separate ticket.

7. **No `IngestService` wiring or SF cutover.** Wave S contract
   explicitly forbids Salesforce / CareStack behavior changes. The
   `SfLeadIngestService._resolve_person` cutover and the
   `tests/ingest/test_sf_lead_service.py` rewrite are documented in
   `reports/R2.md` (PR B) and assigned to S2 / the next wave.
