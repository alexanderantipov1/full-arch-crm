## Current Salesforce Ingest Flow Summary

### Entry point

`apps/api/routers/integrations.py::pull_recent` (POST
`/integrations/salesforce/pull-recent`) calls
`SfLeadIngestService.pull_recent(tenant_id, limit)` (one and only worker
trigger today; no `apps/worker/jobs/ingest_salesforce.py` yet).

### Per-record loop (`SfLeadIngestService.pull_recent`)

For each SOQL record returned by
`SELECT Id, FirstName, LastName, Email, Phone, Company, LeadSource,
Status, CreatedDate FROM Lead ORDER BY CreatedDate DESC LIMIT {N}`:

1. **Capture raw event** — `IngestService.capture(tenant_id,
   RawEventIn(source="salesforce", event_type="lead.pull",
   external_id=sf_id, received_at=now(), payload=record))`. Verbatim,
   before any mapping. Returns the persisted `RawEvent` row but the
   service does NOT keep the row id today.
2. **Resolve person via `_resolve_person(tenant_id, sf_id, record)`**
   (the hidden ENG-100 dedupe ladder):
   - **(a) Re-pull short-circuit.** Calls the protected
     `self._identity._repo.find_source_link(tenant_id, "salesforce",
     "lead", sf_id)`. If found, returns
     `(IdentityService.get_person(tenant_id, link.person_uid), False)`.
     Note: this is the only direct cross-service-into-repo reach in the
     pipeline.
   - **(b) Email reactivation.** If `record["Email"]` present, calls
     `IdentityService.resolve_by_email(...)`. On hit, calls
     `IdentityService.add_source_link(...)` and returns
     `(matched, True)`. `ValidationError` from `normalise_email` is
     swallowed (becomes a no-match, never a crash).
   - **(c) Phone reactivation.** Same as (b) but for
     `record["Phone"]` via `resolve_by_phone`.
   - **(d) Brand-new person fallback.** Builds `PersonIn(...)` with
     name + email/phone identifiers and calls
     `IdentityService.resolve_or_create_person(tenant_id, "salesforce",
     "lead", sf_id, hints=...)`. Returns `(result.person, False)`.
3. **Upsert ops.lead** —
   `OpsService.upsert_lead(tenant_id, person_uid=person.id, raw=record,
   provider_metadata={"sf_lead_id": sf_id, "is_reactivation":
   is_reactivation, "sf_created_at": ..., "company": ...})`.
4. **Build the DTO** — `_to_dto_from_record(...)`. The function reads
   `email` / `phone` from the verbatim SOQL record (NOT from
   `person.identifiers`) to avoid an async lazy-load.

## Proposed ENG-185 Cutover Steps

The cutover is **a single follow-up wave after S1 lands and Codex
reviews it**. It should be planned as one self-contained PR with two
write scopes that don't overlap with each other:

### Step 1 — Capture hint id from `IngestService.capture`

Today `SfLeadIngestService.pull_recent` calls
`self._ingest.capture(...)` and discards the returned `RawEvent`. The
cutover needs the row id so the hint can FK back to the event. Smallest
change: bind the return value, then use `raw_event.id` for the hint
capture.

No new method needed on `IngestService`. The `capture` return value
is already there.

### Step 2 — Capture the normalized person hint

After step 1, call (positions matter):

```python
hint = await self._ingest.capture_normalized_person_hint(
    tenant_id,
    NormalizedPersonHintIn(
        raw_event_id=raw_event.id,
        source_system="salesforce",
        source_kind="lead",
        source_id=sf_id,
        observed_at=datetime.now(UTC),
        given_name=record.get("FirstName"),
        family_name=record.get("LastName"),
        display_name=_join_name(
            record.get("FirstName"), record.get("LastName")
        ),
        email=record.get("Email"),
        phone=record.get("Phone"),
        # payload_sha256 / quality_flags / meta intentionally omitted —
        # the R1 service computes hint_hash from normalized features
        # and records the missing/invalid flags itself.
    ),
)
```

Notes:

- `observed_at` should match the SF `CreatedDate` if we want a
  provider-side timestamp. R1's DTO requires `datetime`, but the SF
  record carries a string ISO-8601. Two options:
  - **Option A (recommended):** pass `datetime.now(UTC)` to match the
    `received_at` semantics of the raw event. Simple, deterministic,
    matches local capture timing.
  - **Option B:** parse `record["CreatedDate"]` and pass it. Requires
    a small ISO-8601 parser; gives a more provider-accurate
    "observed" timestamp. Defer to a follow-up; not needed for
    ENG-185 acceptance.
- `payload_sha256` is optional. R1 left a slot for it but the cutover
  PR does not need it; replay-protection is already covered by the
  raw-event row.
- R1's service refuses PHI-looking keys in `meta`/`quality_flags`.
  We pass empty dicts (defaults) so this is a non-issue.

### Step 3 — Adapt the R1 hint DTO to S1's `MatchHintIn` and call the policy

S1's `tasks/S1.md` specifies the identity-owned input DTO:

```text
MatchHintIn: hint_id, source_system, source_kind, source_id, given_name,
             family_name, display_name, email_normalized,
             phone_normalized, quality_flags, meta
```

The adapter is in `sf_lead_service.py` (ingest is allowed to know about
both DTO shapes; identity must not import ingest). The mapping:

```python
match_hint = MatchHintIn(
    hint_id=hint.id,
    source_system=hint.source_system,
    source_kind=hint.source_kind,
    source_id=hint.source_id,
    given_name=hint.given_name,
    family_name=hint.family_name,
    display_name=hint.display_name,
    email_normalized=hint.email_normalized,
    phone_normalized=hint.phone_normalized,
    quality_flags=hint.quality_flags,
    meta=hint.meta,
)
result: ResolveFromHintResult = (
    await self._identity.resolve_or_create_from_hint(
        tenant_id, match_hint
    )
)
person = await self._identity.get_person(
    tenant_id, PersonUID(result.person_uid)
)
is_reactivation = result.was_existing_person_match
```

### Step 4 — Map `ResolveFromHintResult` → `ops.lead.extra.is_reactivation`

`tasks/S1.md` requires the DTO to expose:

- `was_existing_person_match` — `True` only when tier 1 auto-accept
  matched an existing person.
- `was_source_link_recapture` — `True` for tier 0 exact source-link
  recapture (today's tier (a)).
- `was_new_person` — `True` for tiers 2 and fallback when a new
  person row is created.
- `open_candidate` — `True` only when a `MatchCandidate(status="open")`
  was written (today's "ambiguous" case has no Phase-1 equivalent).

Mapping rule (preserve today's semantics):

| Result flag                       | `is_reactivation` |
|-----------------------------------|-------------------|
| `was_source_link_recapture=True`  | `False` (re-pull)|
| `was_existing_person_match=True`  | `True`           |
| `open_candidate=True` (tier 2)    | `False`          |
| `was_new_person=True` (fallback)  | `False`          |

The new entry point makes `is_reactivation` a single boolean derived
from `result.was_existing_person_match`. `was_source_link_recapture`
naturally implies `was_existing_person_match=False`, so a one-line
mapping suffices:

```python
is_reactivation = bool(result.was_existing_person_match)
```

No `ops.lead.extra` schema change needed; the field stays a bool on the
existing JSONB column.

### Step 5 — Rewrite the SF service `_resolve_person` (or fold it inline)

After the cutover, the ladder collapses to one identity call. Two
acceptable shapes:

- **(i)** Keep `_resolve_person` as a private helper that returns
  `(person, is_reactivation)`. Body shrinks to: build the hint DTO,
  call `capture_normalized_person_hint`, build `MatchHintIn`, call
  `resolve_or_create_from_hint`, fetch the person, return.
- **(ii)** Inline the new flow into `pull_recent` since it is now
  short. Reduces indirection; the helper added no value once the
  ladder was outsourced.

**Recommendation: (i).** Keeps the loop body small and keeps the
SF-specific record→hint mapping in one named function.

### Step 6 — Drop the four obsolete identity-side calls

After the cutover, `SfLeadIngestService` no longer calls:

- `self._identity._repo.find_source_link(...)` (private-reach gone)
- `self._identity.resolve_by_email(...)`
- `self._identity.resolve_by_phone(...)`
- `self._identity.add_source_link(...)`
- `self._identity.resolve_or_create_person(...)`

It still calls `self._identity.get_person(...)` (to resolve the
returned `person_uid` into a row for the DTO and for ops.upsert_lead).

### Optional Step 7 (do NOT include in the cutover PR) — FK migration

Q1 explicitly omitted the DB-level FK `identity.match_candidate.hint_id`
→ `ingest.normalized_person_hint.id` because R1 hadn't shipped. With R1
in main, a small additive migration could add it. **Not part of the
cutover PR.** Keep it as a follow-up ticket so the cutover does not
introduce a migration; Wave S is migration-free by mission goal.

## Risks

### PHI / raw payload

- The identity service must never receive the raw SOQL record. The
  cutover passes a `MatchHintIn` constructed from the R1
  `NormalizedPersonHintOut`; no `record` (the verbatim dict) reaches
  identity. The route and ops layers also keep their existing payload
  fences.
- R1's `capture_normalized_person_hint` already normalizes email/phone
  service-side and refuses PHI keys in `meta`/`quality_flags`. The SF
  cutover passes empty `meta` / `quality_flags`, so the R1 deny-list
  is a defense-in-depth guard, not the primary control.
- Q1's `add_match_candidate` recursive PHI deny-list still guards the
  ledger row that S1 writes. The cutover relies on this guard without
  reimplementing it.

### Tenant isolation

- The SF service forwards `tenant_id` into every identity / ingest
  call (it already does — `_resolve_person` and `pull_recent` both
  take `tenant_id`).
- S1's `resolve_or_create_from_hint(tenant_id, hint)` must filter
  candidate lookup through `for_tenant(...)` — `tasks/S1.md` calls
  this out explicitly. The cutover does NOT need additional tenant
  enforcement at the SF layer; the principal-derived tenant is the
  same tenant id that R1 captures the hint into and that S1 reads
  candidates from.

### Idempotency

For a repeated pull of the same `(salesforce, lead, sf_id)`:

- `IngestService.capture(...)` — already idempotent at the
  application level via the upstream tenant + `external_id` shape;
  duplicate raw events are accepted as today (no behavior change).
- `IngestService.capture_normalized_person_hint(...)` — R1 enforces
  `UNIQUE (tenant_id, raw_event_id)`. Since each re-pull writes a
  **new** `raw_event` row, each re-pull also creates a new hint row.
  This is acceptable for ENG-185: the hint is per-event, not per
  external lead. S1's source-link recapture branch reads the
  existing source link from the new hint, so re-pull still resolves
  to the same person without writing a new match candidate.
- `MatchCandidate` — the partial-unique
  `uq_match_candidate_hint_candidate_active` guard (Q1) prevents two
  active rows for the same `(tenant_id, hint_id, candidate_person_uid)`
  triple. Source-link recapture writes no candidate at all.
- `SourceLink` — `uq_source_link_external` (existing) prevents
  duplicate `(source_system, source_kind, source_id)` rows; tier 0
  bumps `last_seen_at` via `touch_source_link`.

**Cutover assertion in tests:** the re-pull test must mock
`resolve_or_create_from_hint` to return
`was_source_link_recapture=True` and assert no second call to
`capture_normalized_person_hint` causes a `MatchCandidate` to be
written (verified at the identity layer, not here — the SF test only
asserts that the result is mapped to `is_reactivation=False`).

### Open candidates not blocking the pull

- `tasks/S1.md` tier 2 requires `resolve_or_create_from_hint` to
  return a usable `person_uid` even when status is `open` (it creates
  a new source-linked person). The cutover must therefore treat the
  result like any other branch: always reach the `ops.upsert_lead`
  call. The test
  `_ambiguous_email_only_creates_new_person_and_open_candidate`
  guards this.
- The SF service does NOT need to read the `open_candidate` flag
  back into `ops.lead.extra` for Phase 1. If product later wants the
  operator UI to surface "match pending", a small additive write
  into `ops.lead.extra.open_candidate = True` could happen — but
  that is out of scope for the cutover.

### `is_reactivation` semantics preservation

- Today's tiers (b) and (c) → `True`. After cutover, tier 1
  auto-accept → `True`. Same business meaning ("we matched an
  existing person via cross-record evidence").
- Today's tier (a) → `False`. After cutover, tier 0 source-link
  recapture → `False`. Same.
- Today's tier (d) → `False`. After cutover, fallback new-person
  → `False`. Same.
- Today has no tier-2 ambiguous case. After cutover, tier 2 →
  `False`. New semantics, documented in the cutover PR description.
  The dashboard already treats `is_reactivation` as advisory, not a
  hard fact, so a `False` for the open case is the conservative
  default.

### Backward compatibility of ops.lead

- `ops.lead` row identity is `person_uid`. The cutover does not
  change which `person_uid` resolves for a given SF record — tier
  0/1/2 + fallback collectively cover every branch the old ladder
  did, with the same resolution decisions for unambiguous cases.
- `provider_metadata` keys are unchanged. The new `is_reactivation`
  comes from the new result DTO, not from `extra`.
