## Surface mapping — what is actually returning 409

`409 Conflict` on `/integrations/salesforce/{recent-leads, pull-recent,
sync, lead/.../raw}` is the platform-error envelope translation of
`SfNotConnectedError` (`packages/integrations/salesforce/exceptions.py`).
It is raised in two places that are reachable from the operator UI:

1. `apps/api/dependencies.py::_build_salesforce_client` — when
   `IntegrationCredentialService.read_for(tenant_id, "salesforce",
   "oauth_token")` raises `NoCredentialError`. This is the
   `recent-leads`, `pull-recent`, `lead/{id}/raw` and `sync` 409s.
2. `packages/integrations/salesforce/client.py::SfClient.{soql,
   get_object}` — on a SF 401 with no `refresh_token` or after a
   refresh that itself returns `invalid_grant`. The serving image
   would log `sf.token.refresh_failed` in that case; none observed.

Combined with "no `sf.oauth.token_persisted` since 2026-05-16 23:03",
the conclusion is that the current `tenant.integration_credential`
state has **no active `(salesforce, oauth_token)` row** and the OAuth
round-trip is not completing on the operator's browser side.

## Likely root cause(s)

Evidence-supported:

1. **OAuth callback never lands on the API.** Between 2026-05-16
   23:15:06 (last successful flow's final `connect/start`) and now,
   every `POST /integrations/salesforce/connect/start` has returned
   200 with an authorize URL, but **no subsequent
   `GET /integrations/salesforce/callback`** has been recorded for
   the same tenant. Without callback execution, no
   `(salesforce, oauth_token)` row is ever (re-)written, and every
   downstream route therefore raises `SfNotConnectedError → 409`.

2. **Operator UI is wedged in a reconnect loop because of (1).** The
   integration page (`fusion-web-00029-zdf`) calls `recent-leads` /
   `pull-recent` on mount and on user action, both of which now 409
   on the dependency. The Connect button mints a fresh authorize URL
   each click but the browser-side round-trip never returns. The
   loop is operator-visible exactly as the user reported.

Hypothesis-only (cannot confirm from read-only seat):

3. **Salesforce-side OAuth error before the consent screen.** Most
   parsimonious explanation for "connect_start 200 → no callback at
   all". Candidate causes:
   - The DB-backed `(salesforce, api_key)` payload `callback_url`
     does not match the Connected App's whitelisted callback URL on
     Salesforce. `load_client_config` in `packages/integrations/
     salesforce/oauth.py:108-111` resolves `callback_url` as
     `settings.salesforce_callback_url or payload.callback_url`. On
     this revision `SALESFORCE_CALLBACK_URL` env is **unset**, so the
     only source is the DB row. If that row drifted from the SF
     admin-side allowlist (e.g. an admin rotated the connected app
     or someone re-seeded the row with a `localhost` / preview URL),
     Salesforce would refuse the request with `redirect_uri_mismatch`
     on its own domain — the operator would see an SF error page and
     no redirect back would fire.
   - The DB-backed `(salesforce, api_key)` payload `client_id` /
     `client_secret` are stale relative to the Salesforce Connected
     App. Same outward symptom as above (`invalid_client_id`).
   - A popup/iframe blocker on the operator's browser closed the SF
     consent window before completion.

4. **PKCE / state cookie loss across the round-trip.** Less likely:
   if the cookies set by `connect_start` were not received back by
   the callback, the callback handler would still execute and emit
   `sf.oauth.callback_error=state_mismatch` or `missing_pkce_cookie`.
   We see zero callback events, so this is ruled out.

5. **Inbound routing change that drops `/integrations/salesforce/
   callback` before it reaches Cloud Run.** The path
   `/api/integrations/salesforce/callback` is documented as living
   in `apps/api/routers/` (per `reference_prod_routing_split`
   memory). No request lines for `/callback` appear in the API
   service logs for the broken session, but they DID appear for
   yesterday's working session against the **same revision**, so the
   route is still wired through the load balancer / IAP. This makes
   "routing regression" unlikely; the round-trip is dying on the
   Salesforce side.

(3) is the most evidence-consistent root cause. (1) and (2) are
direct observations, not hypotheses.
