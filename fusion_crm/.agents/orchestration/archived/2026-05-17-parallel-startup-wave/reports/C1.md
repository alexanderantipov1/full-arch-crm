**ENG-168 starts first**, but only the design slices (W1-T2, W1-T3 below) until ENG-165 / B1 closes. Reasoning:

1. ENG-168 is the upstream of both ENG-166 and ENG-167 — every SMS those features eventually send must go through the controlled-send + consent/opt-out boundary built in ENG-168.

4. ENG-166 and ENG-167 require ENG-169 (live-runtime ADR) before their subscriber/polling runtime can be deployed — those features parallelize cleanly only after Wave 1 unblocks them.

ENG-167 starts second because CareStack appointment polling can be designed and unit-tested against the existing `CareStackClient` without new runtime; ENG-166 starts third because its CDC subscriber/outbound-message receiver decision sits behind ENG-169.

## High-Risk Areas

### PHI boundaries

- CareStack appointment payloads contain PHI (`patientId`, `notes`, status text). Every CareStack-touching task in this plan must:
  - never log raw payload bodies (logs carry only `appointment_id`, `lastUpdatedOn`, `tenant_id`, `sync_run_id`);
  - write to `phi.*` only through `PhiService`;
  - keep `consultation_scheduled_context` to safe operational fields (status, location, appointment time, treatment interest if non-clinical, financing concern, follow-up due) — clinical notes stay in `phi`.
- Salesforce Leads are not formally PHI in this org but the same logging discipline applies (no name/phone/email in logs).
- SMS bodies are operationally sensitive even if not PHI — store only hashed recipient + template key + provider message id in audit. Bodies belong in `interaction.event_content` (M3, not now).

### Tenant credentials

- B1 must close first. No Wave-2 work touches `packages/tenant/credential_service.py` directly.
- Twilio credential payload shape is decided in W1-T2 ADR and reviewed by Codex before W2-T1 implements `from_credential`.
- Plaintext credential values never appear in audit `extra` — mirror the existing tenant audit sanitiser.
- `INTERNAL_CREDENTIAL_TOKEN` is the only acceptable cross-process resolver path; do not invent a new resolver.

### SMS consent / opt-out

- STOP/UNSTOP/HELP keywords must be processed before any reply is sent (Twilio inbound webhook → suppression upsert). Even though inbound webhook is out of scope for ENG-168 MVP, the suppression schema must be future-proof for this.
- Quiet-hours policy is per-tenant via `tenant.setting` key `sms.quiet_hours` (new key — document in W1-T3 ADR).
- Allowed-template list lives in `sms.allowed_template` keyed by `(tenant_id, template_key)`. Workflow trigger must pass `template_key`; raw bodies are never allowed at the workflow boundary.
- Duplicate-send guard via composite idempotency key `(tenant_id, person_uid, template_key, source_event_id)`.

### Cloud Run / worker runtime decisions

- ENG-169 ADR is the gate. Until it lands:
  - HTTP receivers (Wave 2 SF intake leg) are safe — they are normal request/response Cloud Run Services.
  - Cron-style polling (Wave 2 CareStack appointment poller) belongs in Cloud Run Jobs, NOT a new Service.
  - CDC streaming subscribers, long-running queue workers, and any new long-running runtime must wait for ENG-169.
