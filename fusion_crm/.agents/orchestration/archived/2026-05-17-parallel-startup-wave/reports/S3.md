### Tenant Isolation Risks

The new repository helper(s) S1 introduces for candidate lookup must
filter by tenant_id on every join:

- A naive `select(Person).join(PersonIdentifier).where(value=email)` is
  WRONG — it can return cross-tenant rows that share a normalised email.
- Correct pattern (mirrors existing `find_identifier(...)`):
  ```python
  for_tenant(select(Person).join(PersonIdentifier), tenant_id, Person).where(
      PersonIdentifier.tenant_id == tenant_id,
      PersonIdentifier.kind == "email",
      PersonIdentifier.value == email,
  )
  ```
  (or the equivalent with `for_tenant(..., PersonIdentifier)` plus a
  Person tenant filter).
- The auto-accept path must also re-check the candidate person belongs
  to the current tenant. Q1 already enforces this in
  `add_match_candidate(...)` via `_require_person_in_tenant(...)`. The
  new entry point should leverage the same helper for any
  `source_person_uid` / `candidate_person_uid` / `accepted_person_uid`
  it constructs.
- Required test: a hint that matches a person record in a DIFFERENT
  tenant by email/phone must NOT auto-accept. The new helper returns an
  empty candidate list and the entry point falls back to "new person".

### Idempotency Risks

1. **Source-link recapture (tier 0)** must NOT write a match candidate.
   The contract is "exact `(source_system, source_kind, source_id)`
   recapture leaves the ledger untouched". Existing `find_source_link`
   + `touch_source_link` already covers the lookup + bump pattern.
   Test should assert that `_repo.add_match_candidate` is never called
   on a re-pull of the same external triple.
2. **Auto-accept (tier 1) replay** must be guarded by the existing
   partial unique
   `uq_match_candidate_hint_candidate_active` on
   `(tenant_id, hint_id, candidate_person_uid)` for active statuses.
   The service should first call `find_active_hint_candidate(...)` and
   return the existing row when present, rather than relying on the DB
   to raise an integrity error.
3. **Source link write inside auto-accept** must run BEFORE the
   `MatchCandidate` insert in the same unit of work (R2 §"Source-link
   behavior"). Reasoning: if the candidate insert succeeds but the
   source_link insert was deferred and fails on commit, the partial
   unique guard would still treat the auto-accepted row as active on
   the next replay, blocking the recovery insert. Order: source_link
   first → match_candidate second → caller commits.
4. **Open candidate (tier 2) replay** must reuse the existing open row
   on re-call rather than creating a parallel open row. The partial
   unique `uq_match_candidate_open_pair` (sorted-pair key on
   `(tenant_id, person_pair_key)` for `status='open'`) covers the
   existing-pair case. For hint-only opens (no existing person pair),
   the `uq_match_candidate_hint_candidate_active` guard is the relevant
   one. Service should call `find_active_hint_candidate(...)` first.
5. **New-person fallback** must produce a `source_link` exactly once per
   `(tenant_id, source_system, source_kind, source_id)` triple. Existing
   `uq_source_link_external` partial unique covers this. The fallback
   path should call the existing `resolve_or_create_person(...)` so the
   guarantees stack on top of already-tested behavior, rather than
   re-implementing person + source_link creation.

### PHI / Raw-Payload Guard

- `_FORBIDDEN_EVIDENCE_KEYS` and `_reject_phi_keys` already live in
  `packages/identity/service.py` (lines 50-129). The new entry point
  must funnel any constructed `evidence` / `conflicts` payload through
  this guard before insert.
- `MatchHintIn.meta` and `MatchHintIn.quality_flags` (mirror of R1's
  `NormalizedPersonHintIn`) must also pass through the same recursive
  deny-list. Defense in depth — even though R1 catches PHI keys upstream
  on the ingest side, identity should re-check at its own boundary so a
  caller that bypasses ingest (e.g. a hand-built `MatchHintIn`) cannot
  smuggle PHI into the ledger.
- Test coverage required: nested deny-list paths
  (`{"meta": {"clinical_notes": "..."}}`) and key casing
  (`"DOB"` vs `"dob"`) — both already tested by Q1; the new tests
  should at minimum touch the same code path with a hint-derived
  evidence dict.
