# Data Intelligence Agent Contract V1

This contract defines the internal Data Intelligence Agent for the Semantic
Context And Analytics Foundation mission. The agent is a planning and discovery
helper for authorized internal builders. It is not a production manager chat
surface and it is not allowed to access the database directly.

## Purpose

The Data Intelligence Agent helps the team understand real local data shape,
source coverage, linkage quality, and semantic gaps so the catalog, query spec,
read models, and workbench are grounded in observed evidence.

## Role

The agent may:

- inspect repository code, DTOs, service contracts, migrations, tests, and
  provider documentation;
- use approved local read-only tools to profile allowlisted data sources;
- summarize distinct values, null rates, linkage rates, source coverage,
  evidence availability, and quality gaps;
- propose semantic mappings and catalog changes;
- produce gap briefs for Orchestrator handoff.

The agent must not:

- read `.env*` files;
- connect to the database directly;
- run raw SQL generated by an LLM;
- write data, run migrations, mutate provider state, or change production
  records;
- dump raw tables or raw provider payloads;
- expose local samples to production manager UI or manager chat;
- bypass service-layer or tool-layer policy checks.

## Approved Tool Boundary

The approved local tooling should be service-owned or tool-owned, not
agent-owned. A future implementation may expose commands such as:

- list approved query registry entries;
- describe allowlisted datasets and fields;
- profile distinct values for allowlisted fields;
- compute null rates and top values;
- compute source-system distribution;
- compute Salesforce-person-CareStack linkage rates;
- sample bounded, masked rows;
- inspect evidence presence for payment, treatment, consultation, and source
  attribution;
- detect missing mappings and candidate semantic values.

All tools must be read-only and deny-by-default.

## Environment And Access Policy

- Local/dev exploratory mode only until production policies are explicitly
  approved.
- Current production users are treated as authorized internal users for this
  mission phase, but this agent remains an internal builder tool.
- Masked local exploratory samples are allowed through approved read-only
  tooling.
- Direct database access by agents remains forbidden.
- The tool implementation owns credentials and access; the agent never reads
  secrets.

## Data Limits

Recommended defaults:

- row sample limit: 25 rows per request;
- maximum profile groups: 100 distinct values per field;
- maximum fields per request: 20;
- maximum date range for profiling: 365 days unless explicitly narrowed by a
  human operator;
- raw payload fields: denied by default;
- clinical-note fields: denied by default;
- full table exports: denied.

## Masking And Redaction

Default masking:

- names: initials or stable pseudonym;
- phone/email: masked with last four characters only when needed;
- dates of birth: denied unless specifically approved through PHI tooling;
- payment amounts: allowed for analytics profiling, but never paired with raw
  provider payloads;
- free text clinical notes: denied;
- raw payload JSON: denied except through explicitly local-dev inspector
  surfaces outside ordinary agent outputs.

## Data Classes

| Data class | Agent access posture |
| --- | --- |
| `ops` | Profile and sample through approved read-only tools. |
| `identity` | Use `person_uid` and masked identifiers only. |
| `integration_metadata` | Source ids and evidence references allowed; raw payloads denied. |
| `billing` | Profile and bounded masked samples allowed for authorized builders. |
| `phi_adjacent` | Allowed only as reviewed metadata with data-class markings. |
| `phi` | Denied for V1 unless a PHI-capable tool lane is explicitly approved. |

## Output Formats

### Field Profile

```yaml
profile_type: field_profile
source: salesforce|carestack|fusion
field: string
data_class: ops|identity|integration_metadata|billing|phi_adjacent|phi
row_count: integer
null_rate: number
top_values:
  - value: string
    count: integer
    normalized_candidate: string|null
warnings:
  - string
```

### Linkage Brief

```yaml
brief_type: linkage_brief
cohort: string
source_window: string
salesforce_count: integer
person_link_count: integer
carestack_link_count: integer
link_rate: number
gap_examples:
  - masked_subject: string
    missing_link: no_person_link|no_carestack_link|both_missing
    source_reference: string
recommendations:
  - string
```

### Semantic Mapping Proposal

```yaml
brief_type: semantic_mapping_proposal
term: string
source_values:
  - source_system: string
    field: string
    observed_value: string
    proposed_normalized_value: string
confidence: low|medium|high
data_class: string
requires_human_review: boolean
questions_supported:
  - Q1
```

### Gap Brief

```yaml
brief_type: gap_brief
gap: string
affected_questions:
  - Q13
  - Q28
severity: low|medium|high
evidence_summary: string
recommended_linear_issue: string|null
blocked_by:
  - string
```

## First-Wave Targets

The agent should focus on gaps that support the current catalog and read-model
work:

- source normalization for Facebook, Google Ads, paid social, and UTM
  combinations;
- Salesforce lead to Fusion person to CareStack patient linkage rates;
- consultation scheduled/completed/no-show status coverage;
- treatment proposed/accepted evidence availability;
- payment received, refund, failed payment, and balance evidence availability;
- owner/provider/location/business-unit field coverage;
- missing source references for accepted treatment and paid lead cohorts.

## Audit And Logging

Every approved tool call should record:

- user or agent session;
- tool name and query id;
- requested dataset, fields, filters, and row limit;
- data classes touched;
- masking/redaction policy applied;
- row count returned;
- whether billing, PHI-adjacent, or PHI was requested;
- timestamp and result status.

## Do-Not-Do Conditions

- Do not build a generic SQL terminal.
- Do not let LLM prompts define production metrics.
- Do not return raw provider payloads in ordinary agent outputs.
- Do not expose local samples to production manager chat.
- Do not treat local exploratory access as production access policy.
- Do not skip services/tools and read repositories or database tables directly.
