# Full Schema Design — Fusion CRM v0.2

> **Status:** draft for review, 2026-04-30
> **Owner:** TBD
> **Scope:** every table, FK, index, status enum, JSONB shape, and lifecycle
> needed to bring the data model from MVP (5 schemas, 8 tables) to the
> doctrine described in `ai_context_workflow_whitepaper.md` and future
> agent-driven workflow doctrine.
> **Out of scope:** runtime HIPAA enforcement (`Principal.can_read_phi()`,
> vendor BAA gating, data-class sentinels) — see
> `docs/WORKFLOW.md` §8 deferred-work list.

This doc is the source for the v0.2 alembic migration. After it is approved,
ORM models in `packages/<domain>/models.py` are generated from it and the
migration is generated by alembic.

Rows or sections explicitly marked "future" are roadmap reservations. They are
not included in the active migration until the phase plan that owns them is
approved.

## 0. Conventions

These apply to every table unless explicitly noted.

- **Primary key** is always `id UUID NOT NULL DEFAULT gen_random_uuid()`. We use
  `IDMixin` from `packages/db/base.py`.
- **Timestamps:** every domain table has `created_at TIMESTAMPTZ NOT NULL DEFAULT
  now()` and `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()` from `TimestampMixin`.
  Audit / append-only tables omit `updated_at`.
- **Foreign keys:** plain UUID columns named `<entity>_id` or `person_uid` for
  the canonical Person reference. `ON DELETE` is `RESTRICT` by default; soft
  deletion is preferred over hard. Where cascade is appropriate it's noted.
- **Polymorphic refs:** when a column references "any entity", we use a pair
  `(<col>_entity_type TEXT, <col>_entity_id UUID)`. No DB-level FK; the
  entity_type discriminates which schema/table.
- **Enums:** stored as `TEXT` with a `CHECK` constraint listing allowed
  values. Python `Literal[...]` types mirror them. We do **not** use Postgres
  `CREATE TYPE ENUM` (rigid for migrations; the CHECK approach is mutable).
- **JSONB:** all flexible fields are `JSONB NOT NULL DEFAULT '{}'::jsonb` or
  `'[]'::jsonb`. Empty object is the "no data" state, not NULL.
- **Append-only tables** (everything in `audit.*`, `ingest.raw_event`,
  `interaction.event`, `context.context_fact` versions): no application
  `UPDATE`/`DELETE`. Production DB role enforces.
- **Multi-tenancy:** every row that is logically "per company" has a
  `company_uid UUID NOT NULL DEFAULT '<GLOBAL>'::uuid`. The constant `GLOBAL`
  is a fixed UUID stored in `packages/core/constants.py`. When `Company`
  becomes a real entity (Phase 10), this column flips from stub to real FK
  with no schema change.
- **Naming:** table names are `snake_case` singular, columns are `snake_case`,
  enum values are `snake_case`. Index naming follows alembic conventions
  (`ix_<table>_<col>` for normal, `uq_<table>_<col>` for unique).

## 2. Schema `identity` — extensions

Live tables (`person`, `person_identifier`) unchanged. New tables below.

### 2.1 `identity.source_link`

Maps a Person to a record in an external system. Like `person_identifier`
but tracks the *origin* of the person row, not just an external id alias.
Useful for "this person was first seen in Salesforce as Lead X".

| Column | Type | Null | Default | Notes |
|---|---|---|---|---|
| id | UUID | NO | gen_random_uuid() | PK |
| person_uid | UUID | NO | — | FK identity.person.id |
| source_system | TEXT | NO | — | 'salesforce' \| 'carestack' \| 'twilio' \| 'vapi' \| 'web_form' \| 'manual' \| 'import' |
| source_kind | TEXT | NO | — | 'lead' \| 'contact' \| 'patient' \| 'caller' \| 'sms_sender' \| 'submitter' |
| source_id | TEXT | YES | — | external row id, if applicable |
| first_seen_at | TIMESTAMPTZ | NO | now() | when this link was first observed |
| last_seen_at | TIMESTAMPTZ | NO | now() | refreshed on every later observation |
| meta | JSONB | NO | '{}' | source-specific |

**Indexes:**
- `ix_source_link_person_uid` on `person_uid`
- `ix_source_link_source` on `(source_system, source_kind)`
- `uq_source_link_external` UNIQUE `(source_system, source_kind, source_id)` WHERE `source_id IS NOT NULL`

### 2.2 `identity.merge_event`

Records when two Persons were merged. Append-only. Never deleted.

| Column | Type | Null | Default | Notes |
|---|---|---|---|---|
| id | UUID | NO | gen_random_uuid() | PK |
| surviving_person_uid | UUID | NO | — | FK identity.person.id; the row that "wins" |
| merged_person_uid | UUID | NO | — | the person_uid that was retired (row stays for history) |
| reason | TEXT | NO | — | 'duplicate_email' \| 'duplicate_phone' \| 'manual' \| 'cross_provider_match' |
| evidence | JSONB | NO | '{}' | what made us decide |
| performed_by_actor_id | UUID | YES | — | FK actor.actor.id; who/what merged |
| occurred_at | TIMESTAMPTZ | NO | now() | |

**Indexes:**
- `ix_merge_event_surviving` on `surviving_person_uid`
- `ix_merge_event_merged` on `merged_person_uid`

**Lifecycle:** append-only; merges are never reversed in the same row — a
reverse merge creates a new row with the previous merge's `surviving_person_uid`
as the new `merged_person_uid`.

## 15. Schema `auth` — new

Authentication and authorization plumbing. Serves three subjects:

1. **Staff users** (humans) — coordinators, doctors, managers; backed by an
   `actor.actor` row of type `human`.
2. **Service clients** — MCP server clients (Claude Code, Codex), CI tooling,
   external services; backed by an `actor.actor` row of type `external_service`
   or `ai`. Authenticated via API keys, not passwords.
3. **Patient portal accounts** — future. Patients who have a portal login;
   backed by `auth.portal_account` (NOT an Actor) linked 1:1 to
   `identity.person`. Schema is reserved now so the patient portal can ship
   later without restructuring.

`credential` and `session` tables are **polymorphic** —
`subject_type IN ('actor', 'portal_account')`. Same machinery serves staff
and patients; their lifecycles and policies differ but the storage shape is
unified.

Runtime enforcement of permissions (capability checks, resource-action
allow/deny) is **deferred** (Phase 8). The `permission_grant` table exists
in v0.2 so the data shape is ready; population and enforcement come later.

### 15.6 Auth flow notes (informational)

- **Staff login:** browser POSTs `{email, password}` → API verifies `auth.credential(subject_type='actor', credential_kind='password')` → issues `auth.session` cookie. MFA (when enabled) verifies a second `credential(credential_kind='mfa_totp')` row before the session is marked usable.
- **MCP client connect:** Claude Code presents `Authorization: Bearer fcrm_<token>`. API hashes, looks up `auth.api_key.token_hash`, resolves to `actor_id`, runs all subsequent operations as that Actor. Each tool call writes `audit.agent_tool_call` keyed by the actor.
- **Patient portal login (future):** same as staff login but `subject_type='portal_account'`. PHI access via `PhiService.*` — runtime gating in Phase 8.

## 17. Migration strategy

**One alembic revision** generates all of the above. Generated via
`make db-revision M="v0.2 strategic schema"`, then:

1. Inspect the generated file for ordering issues (FK across new schemas —
   alembic usually handles this but verify).
2. Verify the migration file lives in `packages/db/alembic/versions/`.
3. Apply on a fresh DB: drop volume, `docker compose up -d postgres`,
   `make db-upgrade`. Confirm `\dn` shows all 13 schemas; `alembic current`
   shows the new head.
4. Apply on the existing live DB (current revision `af6c4e767923`) via
   `make db-upgrade`. Confirm no destructive ops; the migration is purely
   additive (new schemas, new tables, new columns on existing tables — none
   of the existing tables get a column drop).
5. Update `init-schemas.sql` (`CREATE SCHEMA IF NOT EXISTS` for all 8 new),
   `DOMAIN_SCHEMAS` in `alembic/env.py`, `packages/db/registry.py`
   (import all new `models.py`).
6. Update `docs/data-model/CATALOG.md` to mark all `(planned)` rows live and
   add the new kinds.
7. Run `alembic check` — must be clean (no drift between models and DB).

**Rollback:** the migration is additive, so a single `alembic downgrade -1`
drops the whole v0.2 layer cleanly. Acceptable in dev; never in prod.

## 18. Open questions surfaced during this design

These are *not* blockers for the migration — they're choices we'll iterate
on once we touch the use cases:

1. **`actor.actor.role` taxonomy.** Currently free `TEXT`. Should we
   normalize into a separate `role` table when we add UI for editing? Defer.
2. **`workflow.flow_definition.node_template` JSONB shape.** Deliberately
   left flexible — flow templates evolve. Pin the schema after Phase 5.
3. **`segmentation.segment_definition.query_logic` resolver.** Sketched in
   §11.4 but the actual implementation lives in `segmentation/service.py`.
   First pass uses a hand-coded subset; later we may need a real DSL.
4. **`encounter.encounter.location_id`** points to
   `integrations.external_entity` for now (CareStack location). Promote to
   `ops.location` (or `clinic.location`) when we have multiple physical
   locations and the data warrants its own table.
5. **Audit retention.** No retention policy at the DB level. Decided in
   Phase 8 along with HIPAA runtime gating.
6. **PHI summaries split.** `phi.transcript_raw.body` is the long PHI text;
   `interaction.transcript_artifact.inline_text` is meant to be short /
   non-PHI. The boundary is enforced by app code (which writes which).
   Reconsider when we actually have transcripts to test against.
7. **`context.context_fact.contains_phi`** vs `context_type='clinical_phi'`.
   The `contains_phi` flag is the source-of-truth at row level; the
   `context_type` is taxonomy. Both are kept; the flag drives runtime gating
   (when we add it), the type drives queries.
8. **Policy DSL location.** Agent-driven workflow needs policy preflight
   before every tool call. Decide whether policy lives in Python objects,
   `auth.permission_grant`, workflow metadata, or a dedicated policy table.
9. **Workflow timer runner semantics.** `workflow.timer` reserves durable timer
   state, but Phase 5 must choose database polling, Redis/arq scheduled jobs,
   or a hybrid.
10. **Approval queue mutability.** Decide whether `workflow.approval_request`
   status changes are normal updates or append-only events with a projected
   current state.
11. **Staff agent-builder governance.** Define who can create, approve,
   publish, pause, and retire role-scoped agent templates before Phase 10.
